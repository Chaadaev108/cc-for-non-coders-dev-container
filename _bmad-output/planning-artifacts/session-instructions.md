# Инструкции для сессий документирования

## Общие правила

- Каждый этап запускать в **новом контексте** (новый чат)
- Не переходить к следующему этапу, пока не пройден чеклист текущего
- Все артефакты сохраняются на диск и подхватываются следующими агентами автоматически

---

## Этап 1: Document Project

### Запуск

Команда: `/bmad-bmm-document-project`

### Первое сообщение агенту

```
Это brownfield open-source проект — Docker-контейнер рабочей среды
для курса "Claude Code: суперсила для непрограммистов" (Automatica, сезон 1).

Мы — команда программистов-контрибьюторов с публичным форком.
Цель — анализ проекта, документация и PR с улучшениями в upstream.

Нужна подробная документация для двух аудиторий:
1. Программисты — архитектура, кодовая база, точки расширения, data flow
2. Непрограммисты — простые объяснения, что делает каждый компонент

Ключевые компоненты:
- auth-gateway.py (~300 строк) — Python HTTP-прокси с cookie-аутентификацией
- Dockerfile — Ubuntu 22.04, Node.js 22, code-server, Python, LibreOffice, Playwright
- entrypoint.sh — запуск сервисов + генерация .claude/.env
- 5 сессий, 28 демо-проектов, 19 Skills

Запусти initial_scan. Язык документации — русский.
```

### Если агент спросит уточнения

**"Какой тип проекта?"** — infra (Docker-контейнер + сервисы), есть элементы web (auth-gateway как HTTP-прокси). Основной тип: infra.

**"Какой уровень детализации?"** — максимальный. Это документация для опытных программистов + упрощённая версия для непрограммистов.

**"Где сохранять?"** — в `docs/` (настроено как `project_knowledge` в конфиге).

**"Есть ли существующая документация?"** — только `README.md` в корне (226 строк). `docs/` пуст.

### Чеклист завершения

Файлы, которые должны появиться в `docs/`:

- [ ] `index.md` — индекс всех документов
- [ ] `project-scan-report.json` — состояние сканирования (JSON с timestamps, mode, classification)
- [ ] Документ с обзором проекта (назначение, стек, компоненты)
- [ ] Документ с архитектурой (контейнер, порты, сервисы)
- [ ] Документ с описанием кодовой базы (auth-gateway, entrypoint, конфиги)

Что проверить в документах:

- [ ] Язык — русский
- [ ] Упомянуты все три сервиса (auth-gateway, code-server, File Browser)
- [ ] Описана сетевая архитектура (порты 8080, 8081, 9090)
- [ ] Описан механизм аутентификации (cookie, HMAC)
- [ ] Упомянуты переменные окружения из `.env`
- [ ] Описана структура курса (5 сессий, 28 демо, 19 Skills)

## Этап 2: Generate Project Context

### Запуск

Команда: `/bmad-bmm-generate-project-context`

### Первое сообщение агенту

```
Документация проекта создана в docs/ (см. docs/index.md).
Сгенерируй project-context.md на основе документации и анализа кода.

Контекст: это Docker-контейнер для курса по Claude Code.
Мы — контрибьюторы, готовим PR с новой фичей —
системой переключения API-провайдеров.

Фокус project-context.md:
- Паттерны и конвенции кода (Python стиль auth-gateway, bash стиль entrypoint)
- Критические правила: как auth-gateway проксирует запросы, как работает .claude/.env
- Точки расширения для контрибьюторов (BACKENDS dict, env vars, switch-api-key.sh)
- Текущая привязка к Z.AI/GLM — файлы, где захардкожен провайдер
- Ограничения и хрупкие места (что нельзя менять без риска сломать)
```

### Если агент спросит уточнения

**"Какой формат?"** — стандартный `project-context.md` по шаблону BMAD. Компактный, LLM-оптимизированный.

**"Нужно ли сканировать код?"** — да, обязательно. Документация из `docs/` — основа, но агент должен сверить с реальным кодом.

**"Для каких агентов этот контекст?"** — для всех последующих: Tech Writer (Paige), Quick Flow (Barry), любых будущих агентов.

### Чеклист завершения

Файл, который должен появиться:

- [ ] `_bmad-output/project-context.md`

Что проверить в файле:

- [ ] Описаны паттерны кода (стиль Python в auth-gateway, стиль bash в entrypoint)
- [ ] Перечислены все переменные окружения, связанные с API
- [ ] Указаны файлы с захардкоженными значениями провайдера (Dockerfile, docker-compose.yml, claude-settings.json, entrypoint.sh, .env.example)
- [ ] Описан механизм `.claude/.env` (генерируется в entrypoint.sh)
- [ ] Описан `switch-api-key.sh` и его ограничения
- [ ] Есть секция по точкам расширения
- [ ] Файл компактный (не дублирует docs/, а дополняет)

## Этап 3: Tech Writer — документы для двух аудиторий

### Запуск

Загрузить агента: `/bmad-agent-bmm-tech-writer`

### Первое сообщение агенту

```
Прочитай docs/index.md и _bmad-output/project-context.md.

На их основе создай документы для двух аудиторий.
Все документы сохраняй в docs/.

ДЛЯ ПРОГРАММИСТОВ (3 документа):

1. architecture-deep-dive.md — детальная архитектура:
   - Контейнерная архитектура: Docker layers, volumes, network
   - Сетевая модель: порты 8080→8081/9090, прокси, WebSocket relay
   - Data flow auth-gateway.py: жизненный цикл запроса от браузера до code-server
   - Механизм аутентификации: HMAC, cookie, token flow
   - Механизм API-ключей: .claude/.env, switch-api-key.sh
   - Mermaid-диаграммы для каждого flow

2. developer-guide.md — для контрибьюторов:
   - Как поднять окружение (./run.sh, docker compose)
   - Структура кода: что где лежит и зачем
   - Как вносить изменения: workflow контрибьютора
   - Ключевые файлы и их взаимосвязи

3. contributing-guide.md — правила вклада:
   - Стиль кода (Python: auth-gateway, Bash: entrypoint/run)
   - Правила PR
   - Что тестировать при изменениях

ДЛЯ НЕПРОГРАММИСТОВ (2 документа):

4. visual-system-overview.md — как работает система:
   - Mermaid-диаграммы на ПРОСТОМ русском (без жаргона)
   - Минимум узлов, понятные подписи
   - Аналогии из реальной жизни (auth-gateway = охранник, code-server = рабочий стол)
   - Что происходит когда студент заходит на сайт

5. course-map.md — карта курса:
   - Визуализация 5 сессий и прогрессии навыков
   - Обзор 28 демо-проектов по сессиям
   - Обзор 19 Skills по категориям
   - Mermaid-диаграмма навигации по курсу

Язык всех документов — русский.
```

### Если агент спросит уточнения

**"Какой стиль диаграмм?"** — Mermaid в Markdown. Для программистов — детальные (sequence, flowchart, class). Для непрограммистов — только простые flowchart с минимумом узлов и русскими подписями.

**"Нужен ли оглавление?"** — да, в каждом документе.

**"В одном файле или нескольких?"** — 5 отдельных файлов как указано выше. Обновить `docs/index.md` после создания.

### Чеклист завершения

Файлы, которые должны появиться в `docs/`:

- [ ] `architecture-deep-dive.md`
- [ ] `developer-guide.md`
- [ ] `contributing-guide.md`
- [ ] `visual-system-overview.md`
- [ ] `course-map.md`
- [ ] `index.md` — обновлён с новыми документами

Что проверить:

- [ ] Все Mermaid-диаграммы синтаксически корректны (проверить рендеринг на GitHub или в VS Code)
- [ ] architecture-deep-dive.md содержит минимум 4 Mermaid-диаграммы (container, network, auth flow, request lifecycle)
- [ ] visual-system-overview.md не содержит технического жаргона (нет "proxy", "HMAC", "WebSocket" — только простые русские аналогии)
- [ ] developer-guide.md описывает конкретные шаги поднятия окружения
- [ ] course-map.md покрывает все 5 сессий, все 28 демо, все 19 Skills
- [ ] Все документы на русском
- [ ] index.md ссылается на все 5 новых документов
